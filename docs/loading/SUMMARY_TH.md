## 🎯 ภาพรวม

### ปัญหาเริ่มต้น (BUG-007)
ผู้ใช้รายงานว่าไม่สามารถยืนยันโหลด **LD-20260115-0023** ได้ เนื่องจากระบบแจ้งว่า "สต็อกไม่เพียงพอ" แม้ว่าจะเชื่อว่ามีสต็อกอยู่

### การสืบสวน
1. ตรวจสอบ API `/api/mobile/loading/complete`
2. พบว่า API ทำงานถูกต้อง - มีสต็อกขาดจริงๆ ที่ Dispatch
3. ค้นพบว่าเป็นผลจาก **BUG-006** (Pick Confirmation ไม่ได้ปล่อย reservation)

### Root Cause
- **BUG-006**: เอกสารที่ยืนยันหยิบ (Pick Confirmation) **ก่อนวันที่ 18 ม.ค. 2026** ไม่ได้ย้ายสต็อกไป Dispatch/MRTD/PQTD
- สต็อกค้างอยู่ใน Storage Locations
- **Migration 229/230** แก้ไขปัญหาสำหรับเอกสารใหม่เท่านั้น
- เอกสารเก่ายังมีปัญหาอยู่

## 🔍 สิ่งที่เรียนรู้

### 1. BUG-006 vs BUG-007
- **BUG-006**: ปัญหาต้นตอ (Pick Confirmation ไม่ปล่อย reservation)
- **BUG-007**: อาการที่เห็น (Loading Complete ล้มเหลว)
- BUG-007 ไม่ใช่บัคใหม่ แต่เป็นผลจาก BUG-006

### 2. Migration 229/230
- แก้ไข BUG-006 สำหรับเอกสารใหม่
- เอกสารเก่า (ก่อน 18 ม.ค. 2026) ยังมีปัญหา
- ต้องแก้ไขด้วยมือหรือ script

### 3. Stock Flow ที่ถูกต้อง
- **Face Sheet/Picklist**: หยิบเสร็จ → สต็อกไป `Dispatch`
- **Bonus Face Sheet**: หยิบเสร็จ → สต็อกไป `MR01-10/PQ01-10` → แล้วไป `MRTD/PQTD`

### 4. การแก้ไขที่มีประสิทธิภาพ
- วิเคราะห์ปัญหาให้ละเอียด
- สร้างเครื่องมือที่ใช้ซ้ำได้
- ทดสอบด้วย dry-run ก่อนเสมอ
- แก้ไขเฉพาะที่จำเป็น
